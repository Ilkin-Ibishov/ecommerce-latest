# Product & Technical Specification: Whitelabel E-Commerce Platform (v5 - Execution Ready)

> Enhanced from v4. Incorporates 14+ audit fixes and Product Owner decisions.
> PO Decisions: Evolution API (self-hosted), Phone OTP via WhatsApp, PostgreSQL FTS, WhatsApp-only notifications, Discounts/coupons + product merchandising, No shipping zones/returns, Medium scale (100-1000 orders/day), Azerbaijan compliance only.

## 1. Project Overview

A mobile-first, multi-lingual, whitelabel e-commerce progressive web application (PWA). The architecture strictly separates the storefront presentation layer (repurposed from NextMerce) from a custom-built administrative dashboard, all powered by a centralized Supabase backend.

**Key differentiators from v4:**
- Real WhatsApp OTP authentication via Evolution API (not Click-to-Chat deep links)
- PostgreSQL full-text search replacing deleted Algolia
- WhatsApp-only notification pipeline with queue and retry
- Discounts/coupons system with product merchandising
- No shipping zones, no returns workflow
- Inventory concurrency control (optimistic locking)
- Persistent cart with guest-to-auth merge
- Audit log for all admin actions
- Designed for 100-1000 orders/day

## 3. Global Business Rules & Constraints

- **Currency:** Strictly `AZN` across all frontend displays and backend calculations.
- **Payment Method:** Hardcoded to "Pay on Delivery". No third-party payment gateways.
- **Languages:** Azerbaijani (Default - `az`), Russian (`ru`), English (`en`).
- **No Shipping Zones:** All delivery is within Azerbaijan. Flat delivery fee or free-delivery threshold configured in `.env`. No zone-based pricing.
- **No Returns:** Orders move through fulfillment pipeline. "Refused at Delivery" is the closest to a return. No separate returns/RMA workflow.
- **Inventory Allocation:** Stock is deducted **only** upon final order submission, not when items are added to cart. Optimistic locking (`UPDATE ... WHERE stock >= requested_qty`) prevents overselling.
- **Discounts/Coupons:** Admin-created coupon codes with percentage or fixed-amount discounts. Optional min-order threshold. Optional product/category scope. Single-use or multi-use. Expiry dates enforced server-side.
- **Product Merchandising:** Featured products, deal-of-the-day, sale flags, sort-order controls. Admin-driven, not algorithmic.
- **Notifications:** WhatsApp only. All customer-facing notifications (order status, OTP, promo) go through Evolution API. No email, no SMS, no push notifications.
- **Compliance:** Azerbaijan only. GDPR not applicable. Data retention per AZ law. Cookie consent not required (no third-party cookies beyond analytics).

## 4. Authentication: Phone OTP via WhatsApp

### Flow

1. User enters phone number (Azerbaijan format: +994XXXXXXXXX) on login/signup page.
2. Server generates a 6-digit OTP, stores it in `otp_codes` table with 3-minute expiry and 3-attempt limit.
3. Server sends OTP via Evolution API's `sendTextMessage` endpoint to the user's WhatsApp number.
4. User enters OTP on the verification screen.
5. Server verifies OTP (hash match, not expired, attempts < 3).
6. On success: create or fetch user in Supabase Auth (phone-based), issue session.
7. On failure after 3 attempts: invalidate OTP, require resend.

### Database: `otp_codes` Table

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid | PK |
| `phone` | text | +994XXXXXXXXX |
| `code_hash` | text | bcrypt hash of 6-digit code |
| `expires_at` | timestamptz | 3 minutes from creation |
| `attempts` | int | Default 0, max 3 |
| `verified` | boolean | Default false |
| `created_at` | timestamptz | |

### Rate Limiting

- Max 5 OTP requests per phone per hour.
- Max 3 verification attempts per OTP.
- Cooldown of 60 seconds between OTP resends.

### Evolution API Integration

```typescript
// lib/whatsapp/client.ts
const EVOLUTION_API_URL = process.env.EVOLUTION_API_URL;
const EVOLUTION_API_KEY = process.env.EVOLUTION_API_KEY;
const INSTANCE = process.env.EVOLUTION_INSTANCE_NAME;

export async function sendWhatsAppOTP(phone: string, code: string) {
  const response = await fetch(
    `${EVOLUTION_API_URL}/message/sendText/${INSTANCE}`,
    {
      method: 'POST',
      headers: {
        'apikey': EVOLUTION_API_KEY,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        number: phone,
        text: `Your verification code: ${code}\n\nDo not share this code.\n\nExpires in 3 minutes.`,
      }),
    }
  );
  if (!response.ok) throw new Error('Evolution API send failed');
  return response.json();
}
```

### Risk Acknowledgment

Evolution API is unofficial and violates WhatsApp TOS. Number bans are possible. Mitigation:
- Use a dedicated business number, not personal.
- Avoid high-volume broadcast (stick to transactional messages only).
- Monitor for ban signals (messages stop delivering).
- Have a fallback plan: manual Click-to-Chat deep link as degraded mode.

## 5. WhatsApp Notification Pipeline

### Architecture

```
Supabase DB Trigger (on order status change)
  → pg_cron / Supabase Webhook
    → Supabase Edge Function (notification dispatcher)
      → Evolution API (send WhatsApp message)
        → Success: mark notification sent
        → Failure: retry queue (3 attempts, exponential backoff)
```

### Notification Types

| Trigger | Template | Recipient |
|---------|----------|-----------|
| OTP sent | "Your verification code: {code}" | Customer |
| Order confirmed | "Order #{id} confirmed. Total: {total} AZN. Pay on delivery." | Customer |
| Status changed | "Order #{id} status: {status}. Track: {url}" | Customer |
| Order shipped | "Order #{id} shipped! Courier: {courier_info}" | Customer |
| Order delivered | "Order #{id} delivered. Thank you!" | Customer |
| Low stock alert | "Product '{name}' stock: {qty}. Reorder threshold: {threshold}" | Admin |

### Database: `notifications` Table

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid | PK |
| `user_id` | uuid | FK to users (nullable for admin alerts) |
| `type` | text | otp, order_confirmed, status_changed, etc. |
| `channel` | text | Always 'whatsapp' |
| `recipient` | text | Phone number |
| `payload` | jsonb | Template variables |
| `status` | text | pending, sent, failed, retrying |
| `attempts` | int | Default 0 |
| `last_attempt_at` | timestamptz | |
| `created_at` | timestamptz | |
| `sent_at` | timestamptz | |

### Retry Logic

- 3 attempts with exponential backoff (1min, 5min, 15min).
- After 3 failures: mark as `failed`, log for manual review.
- `pg_cron` job runs every minute to pick up `pending` and `retrying` notifications.

## 6. Search: PostgreSQL Full-Text Search

### Schema Addition

```sql
-- Add tsvector columns to products and translations
ALTER TABLE products ADD COLUMN search_vector tsvector;
ALTER TABLE product_translations ADD COLUMN search_vector tsvector;

-- GIN indexes for fast lookup
CREATE INDEX idx_products_search ON products USING GIN(search_vector);
CREATE INDEX idx_product_trans_search ON product_translations USING GIN(search_vector);

-- Triggers to keep search_vector updated
CREATE FUNCTION products_search_vector_update() RETURNS trigger AS $$
BEGIN
  NEW.search_vector :=
    setweight(to_tsvector('simple', coalesce(NEW.sku, '')), 'A');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE FUNCTION product_trans_search_vector_update() RETURNS trigger AS $$
BEGIN
  NEW.search_vector :=
    setweight(to_tsvector('simple', coalesce(NEW.title, '')), 'A') ||
    setweight(to_tsvector('simple', coalesce(NEW.description, '')), 'B');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### Why `simple` Dictionary

PostgreSQL lacks built-in Azerbaijani stemming. `simple` dictionary works for all three languages (az, ru, en) without language-specific stemming. `pg_trgm` handles fuzzy/partial matching for typos.

### API Route

```typescript
// app/api/products/search/route.ts
export async function GET(request: Request) {
  const { searchParams } = new URL(request.url);
  const q = searchParams.get('q');
  const lang = searchParams.get('lang') || 'az';

  const { data, error } = await supabase.rpc('search_products', {
    query_text: q,
    lang_code: lang,
  });
  // ...
}
```

### RPC Function

```sql
CREATE OR REPLACE FUNCTION search_products(query_text text, lang_code text)
RETURNS TABLE(id uuid, title text, description text, price numeric, slug text, rank real) AS $$
BEGIN
  RETURN QUERY
  SELECT p.id, pt.title, pt.description, p.price, p.slug,
         ts_rank(pt.search_vector, plainto_tsquery('simple', query_text)) AS rank
  FROM products p
  JOIN product_translations pt ON pt.product_id = p.id AND pt.lang_code = lang_code
  WHERE pt.search_vector @@ plainto_tsquery('simple', query_text)
     OR pt.title ILIKE '%' || query_text || '%'
  ORDER BY rank DESC
  LIMIT 50;
END;
$$ LANGUAGE plpgsql;
```

## 9. Cart Persistence & Merge

### Guest Cart
- Anonymous users get a `session_id` stored in localStorage.
- `cart_items` table uses `session_id` for guest carts.
- No auth required to add to cart.

### Auth Merge
- On login/OTP verification, all `cart_items` with matching `session_id` are migrated to the user's `user_id`.
- If the user already has cart items for the same product, quantities are summed (not duplicated).
- `session_id` cart items are deleted after merge.

### Cart Expiry
- `pg_cron` job cleans up guest cart items older than 7 days.

## 10. Inventory Concurrency Control

### Optimistic Locking Pattern

```sql
-- Deduct stock on order submission
UPDATE products
SET stock = stock - :quantity
WHERE id = :product_id AND stock >= :quantity;

-- If affected rows = 0, stock was insufficient (race condition caught)
-- Return error to customer: "Sorry, this item just went out of stock"
```

### Cancel & Restock

```sql
-- On order cancellation (admin action)
UPDATE products
SET stock = stock + :quantity
FROM order_items
WHERE order_items.product_id = products.id
  AND order_items.order_id = :order_id;
```

This is atomic. No application-level locking needed for 100-1000 orders/day scale.

## 11. Audit Log

Every admin action that mutates data is logged:

| Action | Entity | What's Logged |
|--------|--------|---------------|
| Create product | products | Full new record |
| Update product | products | Changed fields (old + new) |
| Delete product | products | Full deleted record |
| Change order status | orders | Old status, new status |
| Cancel & restock | orders | Restock quantities |
| Create coupon | coupons | Full new record |
| Moderate comment | comments | Old + new approved state |
| Update RLS | (system) | Policy changes |

Trigger-based: `audit_trigger()` fires on INSERT/UPDATE/DELETE for protected tables, writes to `audit_log` with `current_user_id()`, `inet_client_addr()`, and `row_to_json(OLD/NEW)`.

## 15. Scale Considerations (100-1000 orders/day)

At this scale, the architecture is simple and it should stay simple:

- **Single VPS** (4 vCPU, 8GB RAM) runs Next.js, Evolution API, and Supabase (or use Supabase Cloud).
- **No microservices.** No message queues beyond `pg_cron`. No Redis.
- **Connection pooling:** Supabase handles this. If self-hosted, use PgBouncer.
- **Evolution API:** One instance, one WhatsApp number. Sufficient for 100-1000 orders/day transactional messages.
- **Search:** PostgreSQL FTS handles this volume trivially. No need for Elasticsearch.
- **Images:** Supabase Storage with CDN. Client-side compression keeps file sizes small.
- **If growth exceeds 1000/day:** Add read replicas, move Evolution API to separate container, consider Redis for session cache. But don't pre-optimize.
